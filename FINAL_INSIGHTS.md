## УНИВЕРСАЛЬНЫЕ ПАТТЕРНЫ

### 1. ROI-формула для оценки источников данных

**До реализации можно оценить:**

```
ROI = (Плотность связей) / (Усилия на извлечение)

Где:
- Плотность связей = сколько relations даёт 1 единица данных
- Усилия = структурированность (0-100%)
```

**Примеры:**

| Источник | Плотность | Усилия | ROI | Приоритет |
|----------|-----------|--------|-----|-----------|
| **Calendar** | N×N (высокая) | 1 (90% структурирован) | **Очень высокий** | 1 |
| **Contacts** | 1 (средняя) | 2 (CSV структурирован) | Средний | 2 |
| **Email** | 1-2 (низкая) | 100 (5% структурирован, нужна LLM) | **Очень низкий** | 3 |

**Вывод:** Можно было предсказать Calendar > Email **до реализации**.

---

### 2. Шкала структурированности

**Формальное определение:** 
> Структурированность = предсказуемость схемы данных

| % | Тип | Пример | Нужна ли LLM? |
|---|-----|--------|---------------|
| **100%** | SQL/Database | contacts.db | ❌ Нет |
| **90%** | CSV/ICS | Contacts, Calendar | ❌ Нет |
| **70%** | JSON API | REST APIs | ❌ Нет |
| **50%** | HTML + schema | Structured web pages | ⚠️ Иногда |
| **20%** | Plain HTML | Web scraping | ✅ Да |
| **5%** | Text | Email, documents | ✅ Да (обязательно) |
| **1%** | PDF | Scanned documents | ✅ Да + OCR |

**Правило:** LLM нужна, если структурированность < 70%

---

### 3. Export > API для MVP

**Паттерн работает, если:**
- Task = one-off (не continuous sync)
- Data = personal (не enterprise)
- User count < 10
- **И если export не ограничен**

**Паттерн НЕ работает (API лучше), если:**
1. **Continuous Sync** — нужны данные в real-time
2. **Write Operations** — нужно изменять данные
3. **Scoped Exports** — export скрывает данные (Twitter: export 3200 tweets, API unlimited)

---

### 4. Wide & Deep = Синергия (не trade-off)

**Contacts (Wide)** → Контекст: `a@b.com` → "John Doe, CEO, Initech"  
**Calendar (Deep)** → Связи: John ⟷ Alice через Event X

**Без Wide:** Граф из emails (бесполезно)  
**Без Deep:** Список без связей (бесполезно)  
**Wide + Deep:** Полезный граф ✅

**Вывод:** "Шум" (узлы без связей) не вредит в графовых системах — они просто игнорируются при запросах.

---

### 5. Calendar = Ground Truth (domain-specific)

**НЕ универсально!** Зависит от профессии:

| Профессия | Ground Truth | Почему |
|-----------|--------------|--------|
| **Event-Driven** (кураторы, менеджеры) | Calendar | Встречи = основа работы |
| **Remote-first** (программисты, писатели) | Slack/Email | Асинхронное общение |
| **Public-facing** (CEO, influencer) | LinkedIn/Twitter | Публичные связи |

**Вывод:** Для Ольги (куратор, event-driven) Calendar идеален. Для remote dev — нет.

## ИНСТРУМЕНТ: Zero-Budget Thought Experiment

**Gemini рекомендация:**

> "Перед началом любого проекта, где есть 'дорогой' ресурс (LLM, GPU, API, время человека), примените 'Zero-Budget Thought Experiment': **'Если бы [дорогой ресурс] был запрещен, как бы я решил эту задачу?'**"

**Применение к этому проекту:**

**Вопрос:** "Если бы LLM была запрещена, как построить социальный граф?"  
**Ответ:** "Найти структурированные связи → Calendar!"  
**Результат:** ×50 ROI по сравнению с Email + LLM

**Обобщение:**

| Дорогой ресурс | Zero-Budget вопрос | Найденное решение |
|----------------|-------------------|-------------------|
| LLM | "Как без LLM извлечь связи?" | Структурированные данные (Calendar) |
| GPU | "Как без GPU обучить модель?" | Transfer learning / Tiny models |
| Время человека | "Как без человека собрать данные?" | Автоматические export-ы |
| API credits | "Как без API получить данные?" | Web scraping / Export files |

**Вывод:** Constraint заставляет искать **более эффективное** решение.

## УНИВЕРСАЛЬНЫЕ LESSONS LEARNED

### ✅ Универсальные (применимы везде)

1. **"Начни со структурированных данных"**
   - Всегда ранжируй источники по ROI
   - Структурированные данные: ROI ×100 vs неструктурированные

2. **"Export > API для MVP"**
   - Для one-off personal data tasks
   - Исключения: continuous sync, write ops, scoped exports

3. **"Constraints breed creativity"**
   - Ограничения отсекают "очевидные дорогие решения"
   - Вынуждают искать более эффективные пути

4. **"ROI-Driven Prioritization"**
   - Оценить ROI **до** реализации
   - Плотность связей / Усилия на извлечение

5. **"Zero-Budget Thought Experiment"**
   - Для любого проекта с "дорогим ресурсом"
   - Найти решение без этого ресурса = часто лучшее решение

### ⚠️ Domain-Specific (зависит от контекста)

1. **"Calendar = ground truth"**
   - Только для event-driven профессий
   - Remote-first: Slack/Email лучше

2. **"Wide & Deep синергия"**
   - Только для графовых систем
   - В Excel-таблицах "wide" может быть шумом

3. **"Публичные источники не нужны"**
   - Только для non-public persons
   - Для CEO/influencer: LinkedIn/Twitter критичны

## ФИЛОСОФСКОЕ ОБОБЩЕНИЕ

### Парадокс изобилия

**Наблюдение:** Больше ресурсов ≠ лучше результат

**Этот проект:**
- Budget = $0 → вынудил искать → нашли Calendar → ×50 ROI
- Если Budget = $∞ → пошли по Email → ×1 ROI

**Обобщение:**
> Изобилие ресурсов (бюджет, время, API credits) часто **скрывает** наиболее эффективные решения, потому что "дорогие очевидные пути" становятся доступны.

**Constraint — это фильтр:**
- Отсекает дорогие неоптимальные решения
- Вынуждает искать эффективные пути
- Часто приводит к инновации

**Примеры из истории:**
- **Twitter (140 chars):** SMS constraint → viral innovation
- **Instagram (filters):** Low-quality phone cameras → aesthetic innovation
- **Minecraft (pixel art):** Low GPU power → creative limitation as feature

**Вывод:** Constraints aren't bugs — they're features.
