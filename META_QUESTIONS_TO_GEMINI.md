## УРОВЕНЬ 1: Тактический (Конкретные решения)

### Q1.1: Почему именно ×50, а не ×2?

**Наблюдение:** Calendar оказался эффективнее Email на **порядок** (×50), а не просто "лучше" (×2).

**Вопрос:** Есть ли способ **заранее** оценить этот gap?

**Контекст:**
- Calendar: 1 минута → 5000 relations
- Email (гипотетически): 6 часов → 500-1000 relations
- ROI ratio: ×50

**Почему это важно:**
Если бы мы могли оценить этот gap **до реализации**, мы бы сразу начали с Calendar, а не планировали Email как "приоритет №1".

**Есть ли эвристика для оценки?**
- Структурированность источника? (Calendar = 100%, Email = 5%)
- Плотность связей на единицу данных? (Calendar event = N×N relations, Email = 1-2 mentions)
- Другое?

### Q1.2: Как оценить "достаточность" графа?

**Результат:** 5049 edges (relations)

**Вопрос:** Это **достаточно** для персонального социального графа?

**Контекст:**
- Среднее количество связей в LinkedIn: ~500
- Среднее количество "сильных связей" (Dunbar): ~150
- Наш граф: 5049 co_attended relations

**Метрики достаточности:**
- Плотность графа?
- Coverage (% контактов с хотя бы 1 relation)?
- Average degree?
- Другое?

**Почему это важно:** 
Чтобы понять, когда **остановиться** (не делать Email Pipeline, не искать ещё источники).

## УРОВЕНЬ 2: Стратегический (Архитектурные паттерны)

### Q2.1: Паттерн "Export > API" — обобщение

**Твой вывод:** "Для персональных, one-off задач (MVP) → Export (ICS/CSV) всегда проще."

**Вопрос:** Можно ли **обобщить** этот паттерн?

**Гипотеза:** 
```
IF (task = one-off) AND (data = personal) AND (user_count < 10) 
THEN Export > API
```

**Но есть исключения?**

Например:
- **Twitter/X:** Export ограничен (3200 tweets), API лучше?
- **GitHub:** Export репозиториев сложен, API проще?
- **Spotify:** Export плейлистов — как с ним?

**Вопрос:** Есть ли **правило** для определения "Export vs API" для произвольного источника данных?

### Q2.2: "Структурированность" как метрика

**Инсайт:** "Budget = $0 возможен только с структурированными данными."

**Вопрос:** Как **измерить** структурированность источника **до** начала работы?

**Шкала структурированности (гипотеза):**
```
100% - SQL Database (contacts.db)
 90% - CSV/ICS files (Contacts, Calendar)
 70% - JSON API (REST)
 50% - HTML с schema.org
 20% - Plain HTML
  5% - Email (plain text)
  1% - PDF documents
```

**Вопрос:** Эта шкала корректна? Есть ли формальное определение "структурированности" для целей "нужна ли LLM"?

**Почему это важно:**
Чтобы **заранее** оценить, подходит ли источник для Budget = $0.

### Q2.3: Wide & Deep — trade-off или синергия?

**Результат:** Contacts (Wide) + Calendar (Deep) = оптимум.

**Вопрос:** Это **всегда синергия** или есть **trade-off**?

**Гипотеза trade-off:**
- Contacts (1371 узлов) создали "noise" — много людей без связей
- Может быть, лучше было бы: Calendar only → меньше узлов, но все с relations?

**Counter-argument:**
- Contacts дали **контекст** (email → "John Doe, CEO") без которого Calendar был бы just emails

**Вопрос:** Есть ли сценарии, где Wide **вредит** (noise), а не помогает?

## УРОВЕНЬ 3: Методологический (Процесс работы AI-агента)

### Q3.1: Как найти "Calendar-первый" сразу?

**Что произошло:**
1. Gemini рекомендовал: **Email = приоритет №1**
2. Реализовали Email Pipeline (4 часа работы)
3. Потом обнаружили: **Calendar = 90% результата за 1% усилий**
4. Gemini пересмотрел: "Email = over-engineering"

**Вопрос:** Как AI-агент должен был **изначально** найти решение "Calendar-первый"?

**Возможные подходы:**
1. **Ранжирование источников по ROI** (до реализации)
   - Calendar: структурированность = 90%, плотность relations = high → ROI = ×50
   - Email: структурированность = 5%, плотность = low → ROI = ×2

2. **Эмпирическое тестирование** (Proof of Concept)
   - Потратить 1 час на Calendar PoC
   - Потратить 1 час на Email PoC
   - Сравнить results/hour

3. **Паттерн-matching** (из прошлых проектов)
   - "Для social graph всегда начинай с Calendar"
   - Но у AI нет "прошлых проектов"

**Вопрос:** Какой подход правильный? Или это нормально — обнаружить оптимум через итерацию?

### Q3.2: Роль "критического партнёра" (Gemini)

**Паттерн:**
- **Утром:** Gemini критикует ("Список ≠ Граф"), направляет к Calendar
- **Вечером:** Gemini подтверждает успех, объясняет почему

**Вопрос:** Когда "критический партнёр" **критичен** vs когда **избыточен**?

**Наблюдения:**
- Критика "Список ≠ Граф" была **критична** — без неё остановились бы на Contacts
- Но многие архитектурные детали (Groq vs Claude, SQLite vs Neo4j) не обсуждались с Gemini и всё равно сработали

**Вопрос:** Есть ли **правило** для определения, когда нужен human-in-the-loop / AI-consultant?

**Гипотеза:**
- Нужен на **strategic decisions** (что строим: список vs граф)
- Не нужен на **tactical decisions** (какую библиотеку использовать)

### Q3.3: Budget = $0 как constraint — ограничение или катализатор?

**Два взгляда:**

**Взгляд 1: Ограничение**
- Budget = $0 → не можем использовать Claude API
- → не можем делать Email Pipeline
- → "неполный" граф

**Взгляд 2: Катализатор**
- Budget = $0 → вынуждены искать структурированные источники
- → нашли Calendar (лучше Email)
- → получили **более качественный** результат **быстрее**

**Вопрос:** Budget = $0 породил **лучшее** решение или **ограничил** возможности?

**Контрфактуал:**
Если бы Budget = $∞:
- Начали бы с Email (как планировалось утром)
- Использовали бы Claude API
- Потратили бы 6 часов
- Получили бы граф с ~1500 relations (хуже чем 5000 из Calendar)

**Вывод:** Constraint может быть **feature** (forced innovation)?

## УРОВЕНЬ 4: Философский / Meta

### Q4.1: Природа "социальных связей"

**Обнаружение:** Иерархия источников для социального графа:
```
1. Calendar (co_attended) — 5000 relations ✅
2. Email (mentions) — ~500 relations (гипотеза)
3. LinkedIn (connections) — публичных данных нет
4. Contacts (список) — 0 relations (контекст only)
```

**Вопрос:** Что это говорит о **природе** социальных связей?

**Философия:**
- **Наблюдение 1:** "Связь" = совместное событие (co_attended), а не текстовое упоминание (email) или декларация (LinkedIn "connection")
- **Наблюдение 2:** Календарь — это **ground truth** социальных связей. Email/LinkedIn — это их **отражения**.

**Вопрос:** Это **универсальная истина** или **artifact** конкретного кейса (Ольга)?

Например:
- Для CEO tech компании: LinkedIn > Calendar?
- Для remote worker: Email/Slack > Calendar?
- Для интроверта: Contacts = ground truth (календарь пуст)?

**Что произошло в этом проекте:**

2. ✅ Структурированность: Документация + отчёты

**Но также:**

### Q4.3: Обобщение для других задач

**Этот проект:** "Построить социальный граф контактов"

**Вопрос:** Какие **lessons learned** применимы к **произвольным** задачам?

**Кандидаты на обобщение:**

1. **"Начни со структурированных данных"**
   - Применимо к: Knowledge bases, CRM, любые "graph-building" задачи
   - Исключения?

2. **"Export > API для MVP"**
   - Применимо к: Personal data pipelines, one-off analytics
   - Исключения: Real-time sync, Write operations

3. **"Calendar = ground truth для социального графа"**
   - Применимо к: Personal productivity, relationship management
   - Исключения: Remote-first teams (Slack > Calendar?)

4. **"Budget = $0 → вынужденная инновация"**
   - Применимо к: Bootstrapped startups, personal projects
   - Исключения: Enterprise (где время > деньги)
