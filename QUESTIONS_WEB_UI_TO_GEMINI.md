## Q-DATA: Аномалия в данных — Events в топе relationship_strength

**Наблюдение:**
После исправления Q-G2, в топ-10 по relationship_strength появились **Events**:

```
Топ-10:
1. o.g.rozet@gmail.com: 0.952 (directory)  # Person
2. Ольга Розет: 0.644 (active)            # Person
3. Наталья Логинова: 0.632 (active)       # Person
4. Paris January 2026: 0.628 (active)     # Event (!)
5. Париж 2026: 0.628 (active)             # Event (!)
6. Программа «...»: 0.628 (active)        # Event (!)
7. nsharpanova@...: 0.500 (directory)     # Person
```

**Вопрос:**
Это корректно или проблема?

**Анализ:**

### Сценарий A: Корректно (Events = полноправные entities)
- Events (из Calendar) — это такие же entities, как Person
- Они имеют edges (`co_attended`) с Person
- Высокий relationship_strength = много участников
- **Это нормально**

### Сценарий B: Некорректно (Events не должны быть в топе)
- relationship_strength должен измерять "силу связи **человека**"
- Events не имеют "силы связи" (они — контейнеры)
- Нужно **исключить type='Event'** из расчёта relationship_strength

**Вопрос к Gemini:**
1. Корректно ли, что Events появляются в топе по relationship_strength?
2. Если нет — исключить type='Event' из сценариев Q1, Q5, Q11, Q12?
3. Или Events полезны для некоторых сценариев (например, "самые важные события")?

**Моя гипотеза:**
- Для Q1, Q5 (топ **контактов**) — Events нужно **исключить**
- Но можно добавить отдельный сценарий Q14: "Топ событий" (для Events)
